2025년 11월부터 이듬해 1월까지 석 달 동안, 지점을 여러 곳 운영하는 지도 기반 서비스의 홈페이지 통합 프로젝트에 참여했습니다. 프론트엔드 네 명이 페이지를 나눠 맡았고, 저는 네이버 지도 위에 지점 마커를 띄우고 탭과 필터로 골라 들어가는 지점 찾기 페이지를 담당했습니다. 이 페이지를 직접 눌러 보며 테스트하다가 이상한 동작을 하나 발견했습니다. 탭을 분명히 바꿨는데, 마커를 클릭하면 이전 탭의 경로로 이동했습니다. 마커 클릭 핸들러가 자기가 등록되던 순간의 값을 계속 붙들고 있었기 때문입니다. React에서 말하는 낡은 클로저(Stale Closure) 문제였고, 이 글은 그걸 useRef로 정리한 기록입니다.

## 어떤 문제였나

통합 프로젝트라 대부분의 작업은 기존 기능을 React로 새로 만드는 일이었습니다. 그중 지점 찾기는 제가 자원해서 맡았습니다. 2년쯤 전에 이 페이지의 이전 버전을 jQuery로 만든 사람이 저였기 때문입니다. 화면 구성도 비즈니스 로직도 이미 아는 내용이니, 기술만 React로 갈아입히면 금방 끝나겠다는 게 처음의 가정이었습니다.

막상 옮겨 보니 이 페이지는 상태가 유난히 많이 얽혀 있었습니다. 지점 성격에 따른 탭 세 개, 지역·프로모션·공간 유형·인원 범위 등 여섯 종의 필터, 지도 확대 수준에 따라 마커를 묶어 보여주는 클러스터링, 탭·필터·상세 진입마다 달라지는 지도 중심점 계산, 모바일에서는 드래그로 여닫는 패널까지 한 페이지에 모여 있었습니다. 사용자가 무언가를 건드릴 때마다 지도와 목록과 URL이 함께 움직여야 하는 구조였습니다.

{/* TODO 이미지: 지점 찾기 페이지 전체 화면 캡처 — 지도 + 마커 + 탭 + 필터 + 목록 패널이 한눈에 보이는 컷. 회사·서비스명, 실제 지점명, 로고는 가림 처리 전제 */}

그리고 결정적인 차이가 하나 있었습니다. jQuery 시절에는 값이 필요하면 그 순간의 DOM이나 변수에서 바로 읽어 왔기 때문에, 적어도 이런 식으로 데인 기억은 없습니다. React는 다릅니다. 컴포넌트 함수가 렌더링될 때마다 새로 실행되고, 그 안에서 만들어진 함수들은 그 시점의 상태값을 스냅샷처럼 기억합니다. 여기에 네이버 지도 API라는 React 바깥의 세계가 끼어들면서, jQuery 시절에는 만난 적 없던 종류의 버그가 나왔습니다.

## 탭을 바꿨는데 이전 탭 경로로 갔다

증상은 이렇습니다. 탭이 세 개 있는데 편의상 A 탭과 B 탭으로 부르겠습니다. A 탭에서 마커를 클릭하면 `/place/a/{지점 id}` 상세 페이지로 이동합니다. 여기까지는 정상입니다. 그런데 B 탭으로 전환한 뒤 마커를 클릭해도 여전히 `/place/a/{지점 id}`로 이동했습니다. 화면에는 B 탭의 지점 마커가 떠 있는데, 클릭하면 A 탭의 상세 경로로 갔습니다.

이 버그는 QA 단계가 아니라 제가 개발 중에 페이지를 직접 반복해서 눌러 보다가 발견했습니다. 탭을 바꾸고, 마커를 찍고, 다시 탭을 바꿔 또 찍는 조작을 여러 번 섞어야 드러나는 동작이라 코드만 읽어서는 눈에 띄지 않았습니다.

## 원인은 등록 시점에 갇힌 값이었다

자바스크립트 함수는 자기가 만들어질 때 주변에 있던 변수를 기억합니다. 이걸 클로저(closure)라고 부릅니다. React 함수 컴포넌트에서는 렌더링될 때마다 컴포넌트 함수가 새로 실행되므로, 그 안에서 만들어지는 함수는 그 렌더링 시점의 상태값을 기억합니다.

문제는 네이버 지도가 React 바깥에 있다는 점입니다. 마커 클릭 핸들러는 React가 아니라 지도 API에 등록합니다. 지도와 마커는 생성 비용이 커서 화면이 처음 뜰 때 한 번 만들어 재사용하는 구조였고, 클릭 핸들러도 그때 한 번만 등록됩니다. 단순화하면 이런 모양입니다.

```typescript
// 문제가 있던 구조를 단순화한 예시입니다
useEffect(() => {
  naver.maps.Event.addListener(marker, "click", () => {
    // 이 함수가 기억하는 activeTab은 등록되던 순간의 값입니다
    navigate({ to: `/place/${activeTab}/${branchId}` })
  })
}, []) // 최초 한 번만 등록
```

등록되던 순간의 `activeTab`이 `'a'`였다면, 이 핸들러는 영원히 `'a'`를 기억합니다. 이후 사용자가 탭을 바꿔 React 상태가 `'b'`로 갱신돼도, 이미 지도 API에 넘어가 있는 핸들러 입장에서는 알 길이 없습니다. 등록 시점의 낡은 값을 계속 참조하는 것, 이게 Stale Closure입니다.

가장 먼저 떠오르는 해법은 useEffect 의존성 배열에 `activeTab`을 넣어, 탭이 바뀔 때마다 핸들러를 떼고 다시 등록하는 방식입니다. 동작이야 하겠지만 이 페이지에는 맞지 않습니다. 마커가 클러스터링으로 묶였다 풀리는 구조라 리스너를 다시 붙일 대상 자체가 계속 변하고, 핸들러가 읽어야 하는 상태가 탭 말고도 여럿이라 그걸 전부 의존성으로 걸면 지도 관련 객체를 수시로 해제하고 재생성하게 됩니다.

{/* ⚠️ 미확인: 재등록 방식을 당시 실제로 시도·검토했는지는 원자료에 기록이 없어, 프로젝트 사실이 아닌 일반적인 트레이드오프로 서술했습니다 */}

## useRef로 최신 값을 건네주기

useRef는 리렌더링과 무관하게 늘 같은 자리를 가리키는 상자입니다. 상자 자체는 바뀌지 않고, 안의 내용물(`.current`)만 바꿔 끼울 수 있습니다. 핸들러에게 값을 직접 쥐여 주는 대신 상자를 쥐여 주면, 핸들러는 클릭이 일어난 그 순간에 상자를 열어 최신 값을 읽습니다.

```typescript
const activeTabRef = useRef(activeTab)
const navigateRef = useRef(navigate)

// 렌더링마다 상자 속 내용물을 최신으로 갈아 끼웁니다
useEffect(() => {
  activeTabRef.current = activeTab
  navigateRef.current = navigate
}, [activeTab, navigate])

// 핸들러 등록은 여전히 최초 한 번뿐입니다
onMarkerClick: (id: string) => {
  const currentTab = activeTabRef.current // 클릭 시점의 최신 값
  navigateRef.current({ to: `/place/${currentTab}/${id}` })
}
```

라우팅 함수인 `navigate`까지 ref에 담은 이유도 같습니다. 라우터가 건네주는 navigate 역시 렌더링 시점에 만들어지는 함수라서, 낡은 navigate를 붙들고 있으면 같은 문제가 생길 수 있습니다.

두 방식을 정리하면 이렇습니다.

| | 핸들러 재등록 | useRef 참조 |
|---|---|---|
| 등록 횟수 | 상태가 바뀔 때마다 해제 후 재등록 | 최초 한 번 |
| 값을 읽는 시점 | 등록 시점에 고정 | 이벤트가 일어난 순간 |
| 지도 객체에 주는 영향 | 리스너 해제·재등록 반복 | 없음 |
| 의존성 관리 | 읽는 상태가 늘수록 복잡해짐 | ref 갱신 effect 하나에 집중 |

## 같은 패턴이 필요한 곳이 여섯 군데 넘게 나왔다

한 번 원리를 이해하고 나니 같은 패턴이 필요한 자리가 계속 보였습니다. 활성 탭과 라우팅 함수 외에도 정보창 위치, 직전 정보창 상태, 클러스터 중심 좌표, 저장해 둔 지도 상태, 상세 진입 시 확대 여부, 현재 위치 버튼 활성 여부까지 전부 지도 이벤트가 읽어야 하는 값들이라, 결국 여섯 군데가 넘는 지점에 같은 패턴을 적용했습니다. 하나의 큰 ref 객체에 몰아넣는 대신 관심사별로 ref를 나눠서, 이름만 봐도 무엇을 담는 상자인지 알 수 있게 했습니다.

패턴이 이만큼 퍼지면 문서가 필요해집니다. 팀 온보딩용으로 작성하던 900줄 정도의 기능 명세서에 "Stale Closure 방지 패턴" 섹션을 따로 두고, 코드 예시와 함께 왜 이 패턴이 필요한지를 적었습니다. 이유를 모르고 보면 멀쩡한 상태를 놔두고 굳이 ref를 겹쳐 쓰는 이상한 코드로 보이기 때문입니다.

## 배운 것

첫째, 외부 라이브러리와 React 상태를 연결할 때는 캡처 시점부터 의심해야 합니다. 지도, 차트, 에디터처럼 React 바깥에서 콜백을 보관하는 라이브러리는 React의 리렌더링 주기를 모릅니다. 이번 버그도 React 안에서만 보면 아무 문제가 없는 코드였습니다. 콜백을 외부에 등록하는 코드가 보이면 "이 함수가 읽는 값은 언제의 값인가"를 먼저 묻는 습관이 생겼습니다.

둘째, 이런 버그는 직접 조작해 봐야 드러납니다. 이 프로젝트에서는 구현의 상당 부분에 AI 코딩 도구의 도움을 받았지만, 이 버그는 타입 검사로도 코드 리뷰로도 잡히지 않았고 화면을 실제 사용 흐름대로 여러 번 눌러 보는 과정에서 나왔습니다. 탭을 바꾸는 시점과 마커를 클릭하는 시점 사이의 시간 차가 만들어내는 문제여서, 코드를 정지 화면으로 읽는 방식으로는 보이지 않습니다. 도구가 코드를 빠르게 만들어 줄수록, 사람이 시간 축을 따라 검증하는 역할은 오히려 무거워진다고 느꼈습니다.

셋째, 토대가 되는 라이브러리는 프로젝트 시작 전에 깊이 학습해 둘수록 초기 설계가 깔끔해집니다. 이 버그는 라우터와 React와 지도 API, 세 가지가 만나는 지점에서 나왔습니다. 라우터가 주는 navigate가 렌더링 시점에 만들어지는 함수라는 사실을 처음부터 또렷하게 알고 있었다면, 핸들러를 설계하는 단계에서 ref 패턴을 먼저 깔았을 겁니다. 버그를 잡는 데 쓴 시간의 상당 부분은 사실 학습을 미룬 비용이었습니다.

---

차트 라이브러리의 클릭 콜백, 에디터의 변경 이벤트, WebSocket 메시지 핸들러, setInterval의 타이머 함수까지, React 바깥에 함수를 맡겨 두는 모든 자리에서 같은 문제가 생길 수 있습니다. 등록은 한 번인데 상태는 계속 바뀐다면 그 사이 어딘가에 낡은 값이 숨어 있습니다. 외부 세계에 콜백을 넘길 때는 값 대신 상자를 넘긴다는 원칙 하나로, 이 계열의 버그는 대부분 설계 단계에서 사라집니다. 다음에 React 바깥의 무언가와 연동할 일이 생기면, 코드를 쓰기 전에 이 콜백이 읽는 값이 언제의 값인지부터 물어볼 생각입니다.
