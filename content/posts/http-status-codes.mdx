클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능입니다.

| 상태코드 | 의미 |
|---|---|
| `1xx` (Informational) | 요청이 수신되어 처리 중 |
| `2xx` (Successful) | 요청 정상 처리 |
| `3xx` (Redirection) | 요청을 완료하려면 추가 행동이 필요 |
| `4xx` (Client Error) | 클라이언트 오류, 잘못된 문법 등으로 서버가 요청을 수행할 수 없음 |
| `5xx` (Server Error) | 서버 오류, 서버가 정상 요청을 처리하지 못함 |

### 만약 모르는 상태 코드가 나타나면?

- 클라이언트가 인식할 수 없는 상태코드를 서버가 반환하면, 클라이언트는 상위 상태코드로 해석해서 처리합니다.
- 미래에 새로운 상태 코드가 추가되어도 클라이언트를 변경하지 않아도 됩니다.

---

## 1xx (Informational)

요청이 수신되어 처리 중입니다. 실무에서는 거의 사용되지 않습니다.

---

## 2xx (Successful)

클라이언트의 요청을 성공적으로 처리했음을 나타냅니다.

### 200 OK

요청이 성공했음을 나타냅니다.

### 201 Created

요청이 성공해서 새로운 리소스가 생성되었음을 나타냅니다.

- 생성된 리소스는 응답의 Location 헤더 필드로 식별합니다.

### 202 Accepted

요청이 접수되었으나 처리가 완료되지 않았음을 나타냅니다.

- 배치 처리 같은 곳에서 사용합니다.
- 예) 요청 접수 후 1시간 뒤에 배치 프로세스가 요청을 처리하는 경우

### 204 No Content

서버가 요청을 성공적으로 수행했지만, 응답 페이로드 본문에 보낼 데이터가 없음을 나타냅니다.

- 예) 웹 문서 편집기에서 save 버튼
  - 네이버 카페에서 카페 글쓰기 → 임시 등록을 누르면 `204`가 반환됩니다.

<Video src="/images/http-status-codes/204-no-content-example.mp4" caption="네이버 카페에서 임시저장 시 204 No Content 응답이 반환되는 화면" />

- save 버튼의 결과로 아무 내용이 없어도 됩니다.
- save 버튼을 눌러도 같은 화면을 유지해야 합니다.
- 결과 내용이 없어도 204 메시지만으로 성공을 인식할 수 있습니다.

---

## 3xx (Redirection)

요청을 완료하기 위해 유저 에이전트의 추가 조치가 필요합니다.

- 웹 브라우저는 3xx 응답 결과에 Location 헤더가 있으면 Location 위치로 자동 이동합니다. (리다이렉트)
- 클라이언트와 서버에서 동시에 리다이렉션 처리를 할 경우에는 클라이언트가 좀 더 우선권이 있는 것으로 보입니다.

### 리다이렉션 종류

**영구 리다이렉션**

- 특정 리소스의 URI가 영구적으로 이동합니다.

**일시 리다이렉션**

- 일시적인 변경입니다.
  - 주문 완료 후 주문 내역 화면으로 이동하는 경우

**특수 리다이렉션**

- 결과 대신 캐시를 사용합니다.
  - 캐시가 만료된 것과 관련해서 생성일자나 정보를 확인하는 용도입니다.

---

## 영구 리다이렉션 (301, 308)

리소스의 URI가 영구적으로 이동합니다.

원래의 URL을 사용하지 않고, 검색 엔진 등에서도 변경을 인지합니다.

### 301 Moved Permanently

리다이렉트 시 요청 메서드가 GET으로 변하고, 본문이 제거될 수 있습니다.

### 308 Permanent Redirect

301과 기능은 같습니다.

리다이렉트 시 요청 메서드와 본문을 유지합니다. (처음 POST를 보내면 리다이렉트도 POST를 유지합니다.)

---

## 일시적인 리다이렉션 (302, 303, 307)

리소스의 URI가 일시적으로 변경됩니다.

따라서 검색 엔진 등에서 URL을 변경하면 안 됩니다.

### PRG: Post/Redirect/Get

- POST로 주문 후에 웹 브라우저를 새로고침하면?
  - 새로고침은 다시 요청입니다.
  - 중복 주문이 될 수 있습니다.
- `POST`로 주문 후에 새로고침으로 인한 중복 주문을 방지합니다.
- `POST`로 주문 후에 주문 결과 화면을 `GET` 메서드로 리다이렉트합니다.
- 새로고침해도 결과 화면을 `GET`으로 조회합니다.
- 중복 주문 대신에 결과 화면만 `GET`으로 다시 요청합니다.

### 302 Found

리다이렉트 시 요청 메서드가 GET으로 변하고, 본문이 제거될 수 있습니다.

- 브랜디에서 네이버페이로 구매 시에 302로 처리합니다.

권장은 `303`, `307`이지만, 현실적으로 많은 애플리케이션 라이브러리들이 `302`를 기본 값으로 사용합니다.

### 303 See Other

302와 기능은 같습니다.

리다이렉트 시 요청 메서드가 GET으로 변경됩니다.

### 307 Temporary Redirect

302와 기능은 같습니다.

리다이렉트 시 요청 메서드와 본문을 유지합니다. (요청 메서드를 변경하면 안 됩니다.)

---

## 기타 리다이렉션

### 304 Not Modified

- 캐시를 목적으로 사용합니다.
- 클라이언트에게 리소스가 수정되지 않았음을 알려주며, 클라이언트는 로컬 PC에 저장된 캐시를 재사용합니다.
- 304 응답은 응답에 메시지 바디를 포함하면 안 됩니다.

---

## 4xx (Client Error)

클라이언트 오류입니다.

- 클라이언트의 요청에 잘못된 문법 등으로 서버가 요청을 수행할 수 없습니다.
- **오류의 원인이 클라이언트에 있습니다.**
- 클라이언트가 이미 잘못된 요청, 데이터를 보내고 있기 때문에 똑같은 재시도가 실패합니다.

### 400 Bad Request

클라이언트가 잘못된 요청을 해서 서버가 요청을 처리할 수 없습니다.

- 요청 구문, 메시지 등의 오류입니다.
- 클라이언트는 요청 내용을 다시 검토하고 보내야 합니다.
- 예) 요청 파라미터가 잘못되거나, API 스펙이 맞지 않을 때

### 401 Unauthorized

클라이언트가 해당 리소스에 대한 인증이 필요합니다.

- 인증(Authentication)이 되지 않았습니다.
- 401 오류 발생 시 응답에 `WWW-Authenticate` 헤더와 함께 인증 방법을 설명합니다.
  - 인증(Authentication): 본인이 누구인지 확인 `(로그인)`
  - 인가(Authorization): 권한부여 `(Admin 권한처럼 특정 리소스에 접근할 수 있는 권한)`
  - 오류 메시지가 Unauthorized이지만 인증되지 않음을 뜻합니다. `(이름이 아쉬움)`

### 403 Forbidden

서버가 요청을 이해했지만 승인을 거부했습니다.

- 주로 인증 자격 증명은 있지만, 접근 권한이 불충분한 경우입니다.
  - 예) 관리자가 아닌 사용자가 로그인은 했지만, 관리자 등급의 리소스에 접근하는 경우 (권한 부족)

### 404 Not Found

요청 리소스를 찾을 수 없습니다.

- 요청 리소스가 서버에 없습니다.
- 또는 클라이언트가 권한이 부족한 리소스에 접근할 때, 해당 리소스를 숨기고 싶을 때 사용합니다.

---

## 5xx (Server Error)

서버 오류입니다.

- 서버 문제로 오류가 발생합니다.
- 서버에 문제가 있기 때문에 재시도하면 성공할 수도 있습니다. (복구가 되거나 등)

### 500 Internal Server Error

서버 문제로 오류가 발생했습니다. 애매하면 500 오류를 사용합니다.

- 서버 내부 문제로 오류가 발생합니다.

### 503 Service Unavailable

서비스 이용이 불가합니다.

- 서버가 일시적인 과부하 또는 예정된 작업으로 잠시 요청을 처리할 수 없습니다.
- Retry-After 헤더 필드로 얼마 뒤에 복구되는지 보낼 수도 있습니다.
