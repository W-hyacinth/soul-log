5월 25일 아침, 운영 중인 블로그들에 들어갈 표지와 도식 이미지를 코드로 만들어 보기로 했습니다. 그때까지는 이미지 생성 AI 서비스에 들어가 손으로 하나씩 만들고 있었는데, 글 작성을 자동화하는 계획을 세우다 보니 이미지가 걸림돌이었습니다. 사흘 뒤 파이프라인은 완성됐습니다. 그런데 그 사흘의 마지막에 저를 붙잡은 것은 렌더링 엔진도 폰트도 아니고, 한국어 줄바꿈을 다루는 CSS 한 줄이었습니다.

## 어떤 문제였나

블로그에는 글마다 표지 이미지와 도식이 들어갑니다. 이미지 생성 AI로 만들면 두 가지가 걸립니다. 첫째, 화면에서 손으로 쓰는 서비스라 자동화가 안 됩니다. 유료 이미지 API로 바꾸면 자동화는 되지만 비용이 붙습니다. 둘째, 이게 더 큰 문제인데, 그림을 그려주는 AI는 그림 속 한글 글자를 자주 깨뜨립니다. 획이 뭉개지거나 존재하지 않는 글자가 나옵니다. 표지처럼 글자가 주인공인 이미지에서는 치명적입니다.

그래서 방향을 뒤집었습니다. 글자가 중요한 이미지라면 그리지 말고 조판하자. 카드 레이아웃을 HTML과 CSS로 만들고, 화면 창 없이 백그라운드로 도는 크롬(헤드리스 모드)에게 그 페이지를 열어 스크린샷을 찍게 하면 PNG가 나옵니다. 브라우저는 이미 완성된 한글 조판 엔진이고, 공짜고, 인터넷도 필요 없습니다.

처음에는 이걸로 끝인 줄 알았습니다. 웹 화면 만들던 방식 그대로 옮기면 되겠거니 했습니다. 그런데 화면에 띄우는 웹페이지와 이미지 산출물은 성격이 다릅니다. 웹페이지는 어색한 줄바꿈이 있어도 스크롤과 함께 흘러가지만, 이미지는 그 상태 그대로 굳어서 남습니다. 이 차이를 사흘째 되는 날 제대로 겪었습니다.

## 첫째 날: 한 시간 만에 생성기 네 종

아침 9시 21분에 프로토타입을 보고 싶다고 답했고, 9시 22분에 환경 점검이 돌았습니다. 크롬 실행 파일 경로, 파이썬, 맥에 기본으로 깔린 한글 폰트 목록, 이미지 크기 검증에 쓸 sips 명령어까지 확인이 끝났고, 9시 23분에 첫 스크립트가 만들어졌습니다. 구조는 단순합니다.

1. 파이썬이 제목, 부제, 시리즈명을 받아 HTML 문자열을 조립합니다.
2. 임시 HTML 파일로 저장합니다.
3. 헤드리스 크롬으로 열어 스크린샷을 찍습니다.
4. 임시 파일을 지우고 PNG만 남깁니다.

```bash
chrome --headless --disable-gpu --hide-scrollbars \
  --force-device-scale-factor=2 \
  --window-size=1200,630 \
  --screenshot=cover.png cover.html
```

크기는 링크 미리보기 표준인 1200×630이고, `--force-device-scale-factor=2`를 주면 같은 레이아웃이 두 배 해상도(2400×1260)로 찍혀서 모바일에서도 선명합니다. 폰트는 시스템에 있는 명조(제목)와 고딕(본문)을 그대로 썼습니다. 브라우저가 렌더링하니 글자 깨짐은 원천적으로 없습니다.

![파이썬이 HTML 문자열을 조립하고, 임시 파일로 저장한 뒤, 헤드리스 크롬이 스크린샷으로 캡처해 PNG만 남기는 네 단계 파이프라인 도식](/images/korean-word-break-png-pipeline/pipeline-diagram.png)

첫 샘플이 9시 39분에 나왔고, 이후 한 시간 동안 세로 타임라인, 2단 비교표, 1→N 분기도 생성기가 차례로 추가됐습니다. 블로그마다 다른 색을 입히는 테마 파일과, 글 목록 JSON 하나를 넣으면 여러 타입의 이미지를 일괄 생성하는 배치 스크립트까지 붙여서 10시 22분까지 산출물이 쌓였습니다. 프로토타입을 시작하고 한 시간 남짓 만입니다.

![첫째 날 만든 생성기 네 종의 산출물 유형을 더미 문구로 재현한 썸네일 — 표지 카드, 세로 타임라인, 2단 비교표, 1→N 분기도](/images/korean-word-break-png-pipeline/day1-outputs.png)

다만 이 첫 버전에서 제목 줄바꿈은 기계가 하지 않았습니다. 제목을 `--title1`과 `--title2` 두 조각으로 나눠 받아서 그 사이에 `<br>`을 박는 방식, 그러니까 사람이 미리 줄을 쪼개서 넘기는 방식이었습니다. 나머지 도식들의 글자도 한 줄짜리 짧은 라벨 위주였습니다. 긴 한국어 문장을 브라우저의 자동 줄바꿈에 맡긴 적이 아직 한 번도 없었습니다.

## 둘째 날: 폰트는 내려받으면 됐다

이튿날은 폰트 폭을 넓혔습니다. 시스템 폰트 두 종만으로는 콘텐츠 분위기를 바꿀 수 없어서, 무료로 상업 사용이 가능한 한글 폰트 여섯 종(검은고딕, 도현, 주아 등)을 내려받았습니다. 시스템에 설치하지 않고 `@font-face`, 그러니까 문서 안에서만 쓸 폰트 파일을 지정하는 CSS 규칙으로 이미지 렌더링에만 물렸습니다. 같은 문구를 폰트별로 나란히 렌더한 비교 이미지를 만들어 놓고 보면서 골랐습니다. 그날 만들던 지도 인포그래픽은 여덟 번의 버전을 거치며 제목 폰트가 도현체로 확정됐습니다.

![같은 문구를 무료 상업용 한글 폰트 여섯 종으로 나란히 렌더해 고른 비교표](/images/korean-word-break-png-pipeline/font-compare.png)

이틀이 지나도록 줄바꿈 문제는 나타나지 않았습니다. 지금 돌아보면 안 나타난 게 아니라 나타날 기회가 없었습니다. 산출물의 글자가 전부 짧은 라벨이거나, 사람이 미리 쪼갠 제목이었으니까요.

## 셋째 날: 긴 문장이 들어오자 어절이 잘렸다

27일 저녁, 처음으로 문장이 통째로 들어가는 이미지를 만들게 됐습니다. 블로그 글 하나에 붙일 이미지 세트였는데, 명단을 정리한 표 하나에 항목을 세 그룹으로 나눈 도식 하나, 그리고 인물별 소개 카드 일곱 장이었습니다. 이 카드 아래쪽에 두세 줄짜리 코멘트 문단이 들어갑니다. 브라우저가 알아서 줄을 바꿔야 하는 길이의 한국어가 파이프라인에 처음 들어온 순간입니다.

여기서 한국어 줄바꿈의 기본값 문제가 나옵니다. 브라우저의 기본 설정에서 영어는 공백 단위로만 줄이 바뀌지만, 한중일 문자는 아무 글자 사이에서나 줄이 바뀔 수 있습니다. 그래서 띄어쓰기로 구분되는 말 덩어리(어절)가 중간에서 뚝 잘립니다. 예를 들어 "가능성이"라는 한 어절이 줄 끝에 걸리면 "가능"과 "성이"로 갈라져 두 줄에 나뉘는 식입니다. 화면 웹페이지라면 어색해도 스쳐 지나가지만, 블로그에 올릴 카드 이미지에서는 그 잘린 줄이 영구히 남습니다.

처방은 CSS 한 줄입니다.

```css
body {
  word-break: keep-all;      /* 한글도 어절(공백) 단위로만 줄바꿈 */
  overflow-wrap: break-word; /* 칸보다 긴 한 덩어리는 예외로 자름 */
}
```

`keep-all`은 한중일 문자도 영어처럼 공백 단위로만 줄을 바꾸게 합니다. 함께 넣은 `overflow-wrap: break-word`는 보험입니다. 어절 단위로만 자르다 보면 URL처럼 공백 없이 긴 덩어리가 칸을 뚫고 나갈 수 있는데, 그 경우에만 예외적으로 중간을 자르도록 허용하는 겁니다.

![같은 카드를 keep-all 없이 렌더하면 코멘트 문단의 어절이 줄 끝에서 중간에 갈라지고, keep-all을 적용하면 어절 단위로만 줄이 바뀌는 것을 나란히 비교한 이미지](/images/korean-word-break-png-pipeline/keep-all-before-after.png)

정확히 어떤 출력물에서 처음 잘린 어절을 봤는지는 세션 로그에 남아 있지 않습니다. 대신 결과가 남아 있습니다. 그날 저녁 7시 반과 7시 44분에 저장된 생성기 두 개, 그리고 밤 9시 반을 넘겨 손보던 세 번째 생성기까지, 27일에 만든 세 개에는 모두 `word-break: keep-all`이 들어가 있습니다. 앞의 이틀 동안 만든 스크립트 다섯 개에는 하나도 없습니다. 그리고 밤 8시 무렵의 작업 노트에는 "한글 줄바꿈은 keep-all"이 이날 확정한 표준으로 이미 기록돼 있습니다.

{/* ⚠️ 미확인: 잘린 어절을 실제로 목격한 순간(어떤 카드·어떤 문구였는지)은 세션 로그가 해당 구간을 담고 있지 않아 확인 불가. 본문은 "기록에 없다"고 명시하는 방식으로 처리했으나, 기억나는 구체 장면이 있으면 보강 가능 */}

같은 날 저녁에 한국어 조판 문제를 하나 더 만났습니다. 카드 왼쪽 위에 붙는 알약 모양 라벨(칩)의 세로 중앙 정렬입니다. `inline-flex`에 `align-items: center`를 주면 수치상으로는 정확히 가운데가 되어야 하는데, 한글 폰트는 글자가 차지하는 세로 공간의 위아래 여백이 대칭이 아니어서 눈으로 보면 글자가 떠 보일 수 있습니다. 최종 코드에는 중앙 정렬에 더해 위쪽 패딩을 아래쪽보다 몇 픽셀 크게 주는 눈대중 보정(위 18px, 아래 12px)까지 얹혀 있고, 이 처방도 keep-all과 함께 그날 작업 노트에 표준으로 남았습니다.

![위아래 대칭 패딩만 준 칩은 중심선 기준으로 글자가 위로 떠 보이고, 위쪽 패딩을 아래보다 크게 준 칩은 글자가 중심선에 가운데로 앉는 것을 확대해 비교한 이미지](/images/korean-word-break-png-pipeline/chip-padding-before-after.png)

## 사흘 동안 만난 문제와 처방

| 문제 | 증상 | 처방 |
|---|---|---|
| 이미지 생성 AI의 한글 | 그림 속 글자의 획이 깨지고 뭉개짐 | 그리지 않고 HTML로 조판, 헤드리스 크롬으로 캡처 |
| 어절 중간 잘림 | 긴 문장에서 한 어절이 두 줄로 갈라짐 | `word-break: keep-all` + `overflow-wrap: break-word` |
| 칩 라벨 세로 쏠림 | 수치상 중앙인데 글자가 위로 떠 보임 | `inline-flex` 중앙 정렬 + 위쪽 패딩 보정 |
| 모바일에서 흐릿한 글자 | 1배 해상도 PNG가 축소·확대되며 뭉개짐 | `--force-device-scale-factor=2`로 2배 렌더 |

## 배운 것

첫째, 글자가 중요한 이미지는 그리는 것보다 조판하는 쪽이 확실합니다. 이미지 생성 AI가 한글을 깨뜨리는 건 확률의 문제라 몇 번을 다시 뽑아도 근본적으로 해결되지 않는데, 브라우저 렌더링은 애초에 깨질 여지가 없습니다. 프로토타입을 결정하고 한 시간 만에 생성기 네 종이 나올 만큼 진입 비용도 낮았습니다. 돈이 들지 않고 인터넷 연결도 필요 없다는 점은 자동화 파이프라인에 넣기에 특히 좋은 성질입니다.

둘째, 웹의 기본값은 한국어에 맞춰져 있지 않습니다. `keep-all`은 아는 사람에게는 상식이지만 기본값이 아니라서, 템플릿을 새로 만들 때마다 명시적으로 선언하지 않으면 조용히 빠집니다. 그리고 화면에서는 대충 넘어가던 어색한 줄바꿈이 이미지 산출물에서는 그대로 굳어 결함이 됩니다. 한국어 텍스트를 다루는 HTML 템플릿이라면 용도가 무엇이든 초기화 스타일처럼 깔고 시작하는 게 맞다고 생각하게 됐습니다.

셋째, 같은 시행착오를 반복하지 않으려면 코드에 넣는 것만으로는 부족하고 규칙으로 남겨야 합니다. 27일 저녁에 keep-all과 칩 보정을 작업 노트에 표준으로 적어 뒀는데, 실제로 그날 밤에 새로 만든 세 번째 생성기는 처음부터 keep-all을 갖고 시작했습니다. 스크립트는 복사하다 빠뜨릴 수 있지만, "한글 템플릿은 keep-all부터"라는 문장은 다음 작업의 출발점이 됩니다.

---

이 파이프라인은 이후로도 블로그 글을 만들 때마다 돌고 있습니다. HTML을 이미지로 바꾸는 접근은 링크 미리보기용 OG 이미지, 차트나 표의 공유용 캡처, 리포트 스냅샷처럼 글자가 들어간 이미지 산출물이 필요한 곳이라면 어디든 쓸 수 있습니다. Next.js의 OG 이미지 생성도 결국 같은 원리로 HTML을 그려서 이미지로 내보냅니다. 그리고 그 모든 곳에서, 대상이 한국어라면 시작은 `word-break: keep-all` 한 줄입니다.
