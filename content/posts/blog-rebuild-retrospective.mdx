2월 27일 오후 5시 14분, 이 블로그의 첫 커밋을 올렸습니다. 파일 179개, 19,356줄이 커밋 하나에 통째로 들어갔고 그 안에 글 42편이 담겨 있었습니다. 2020년 가을부터 2024년 가을까지 쓴 글을 Velog, Notion, 그리고 지금은 서비스가 종료된 이전 재직사의 기술 블로그 세 곳에서 모아 온 결과입니다. 옮기는 일 자체는 마크다운 파일 복사에 가까울 거라고 생각했는데, 실제로 시간이 걸린 곳은 코드가 아니었습니다. 원래 서버가 사라져 돌아오지 않는 이미지들, 그리고 배포 20분 만에 드러난 한글 태그 주소 문제였습니다.

![첫 커밋의 git show --stat 출력. 파일 179개, 19,356줄이 커밋 하나에 통째로 들어갔습니다](/images/blog-rebuild-retrospective/first-commit-stat.png)

## 어떤 작업이었나

글이 세 곳에 흩어져 있었습니다. Velog에 발행한 글, Notion에 정리해 두었던 글, 그리고 이전에 다니던 회사의 기술 블로그에 실었던 글입니다. 마지막이 문제였는데, 그 블로그는 서비스가 종료되어 원래 주소로는 접속할 수 없는 상태였습니다.

블로그 자체는 Next.js 15의 App Router로 만들었습니다. 데이터베이스나 별도의 글 관리 도구 없이 글은 전부 MDX 파일(마크다운 안에 컴포넌트를 섞어 쓸 수 있는 형식)로 저장소 안에 두고, 요청이 올 때마다 페이지를 만드는 대신 빌드 시점에 모든 페이지를 미리 만들어 두는 방식(정적 생성)을 택했습니다. MDX 렌더링에는 next-mdx-remote를 서버 컴포넌트 방식으로 썼고, 스타일은 Tailwind CSS v4를 설정 파일 없이 CSS 안에서 디자인 토큰을 정의해 구성했습니다.

처음 가정은 단순했습니다. 어차피 대부분 마크다운이니, 파일을 모아 frontmatter(제목·날짜·태그를 적는 머리말)만 붙이면 끝나겠다는 것이었습니다. 막상 시작해 보니 세 출처의 글이 저마다 완전히 다른 상태였고, 필요한 손질도 제각각이었습니다.

## 출처마다 상태가 달랐다

| 출처 | 글의 상태 | 주로 한 일 |
|---|---|---|
| Velog | 본문과 이미지 모두 살아 있음 | 마크다운 정리, 이미지 주소를 그대로 쓰도록 허용 도메인 등록 |
| Notion 내보내기 | 파일은 온전하지만 블로그 글의 모양이 아님 | 메모체를 경어체로, 한글 파일명을 영문으로, .mov 영상을 .mp4로 |
| 종료된 기술 블로그 | 원래 서버에서 본문과 이미지가 모두 내려감 | Wayback Machine 아카이브에서 복구, 이미지는 일부만 성공 |

Velog가 가장 수월했습니다. 본문도 이미지도 살아 있어서, Velog의 이미지 서버 주소를 그대로 쓰도록 Next.js의 허용 도메인에 등록하는 것으로 해결됐습니다.

Notion에서 내보낸 글은 손이 많이 갔습니다. 블로그에 올리려고 쓴 글이 아니라서 문장이 전부 "~함", "~됨" 같은 메모체였고, 이걸 한 편씩 경어체로 고쳤습니다. 내보내기가 만들어 준 이미지 파일명은 한글이 인코딩된 긴 문자열이라 주소가 깨지기 쉬워서, 내용을 설명하는 영문 이름으로 바꿔 글마다 폴더를 나눠 담았습니다. 화면 녹화 영상은 .mov 파일이었는데, 이 형식은 브라우저 중 사실상 Safari에서만 재생되기 때문에 ffmpeg으로 전부 .mp4로 변환해 넣었습니다.

MDX에서 마크다운 표 문법은 remark-gfm 플러그인(표나 취소선 같은 GitHub 확장 문법을 처리해 주는 부품)이 없으면 HTML 표로 바뀌지 않고 글자 그대로 노출됩니다. 표가 많은 기술 글을 옮기는 작업이라면 처음부터 챙겨야 합니다.

## 이미지는 어디까지 살아 있었나

서비스가 종료된 기술 블로그의 글은 Wayback Machine(인터넷 아카이브가 웹페이지를 주기적으로 저장해 두는 서비스)에 의존했습니다. 다행히 본문 페이지는 저장되어 있었습니다. 문제는 이미지였습니다. Wayback Machine이 페이지를 저장했다고 해서 그 페이지가 불러오는 이미지 파일까지 담아 뒀다는 보장은 없습니다.

특히 걸린 글이 2021년 8월에 실었던 Swiper 적용기였습니다. 스크린샷과 GIF가 많이 들어간 글인데 아카이브된 페이지에서 이미지가 하나도 뜨지 않았습니다. 그래서 CDX API(Wayback Machine에 어떤 주소가 언제 몇 번 저장됐는지 목록으로 돌려주는 조회 창구)로 이미지가 있던 경로를 직접 조회해 봤습니다. 결과는 해당 경로의 저장 기록 0건이었습니다. 이미지 18장이 어디에도 저장되지 않은 채 원래 서버와 함께 사라진 것을 확인했습니다.

![Wayback Machine CDX API로 이미지가 있던 경로를 조회한 터미널. 저장 기록이 한 건도 없어 응답이 비어 있습니다](/images/blog-rebuild-retrospective/cdx-query-empty.png)

복구가 불가능하다는 게 분명해진 뒤에는 방침을 정해야 했습니다. 기억에 의존해 다시 만드는 건 원문 훼손에 가깝다고 판단해서, 이미지가 있던 자리마다 어떤 이미지였는지를 설명하는 문장을 남기는 쪽을 택했습니다. 본문에는 "[이미지 누락: SwiperJS 공식 문서에서 Vue.js 2.x가 지원 대상이 아님을 보여주는 스크린샷]" 같은 표시가 들어갔고, 글 끝에는 원본 이미지가 서버 종료로 복구되지 않아 설명으로 대체했다는 안내를 붙였습니다.

![이식된 Swiper 글 본문. 사라진 이미지 자리에 어떤 이미지였는지 설명하는 이미지 누락 표시가 인용 형태로 들어가 있습니다](/images/blog-rebuild-retrospective/swiper-missing-note.png)

## 배포 20분 뒤에 태그 주소가 깨졌다

오후 5시 14분에 첫 배포를 하고 태그 페이지를 눌러보다가 문제를 발견했습니다. "학습" 태그의 주소가 /tags/학습이 아니라 %ED%95%99%EC%8A%B5 같은 인코딩된 문자열로 노출되고 있었습니다. 원인은 링크를 만들 때 encodeURIComponent로 직접 인코딩한 값에 Next.js App Router가 자동 인코딩을 한 번 더 얹은 것이었습니다. 같은 값을 두 군데서 인코딩하고 있었습니다. 5시 34분, 직접 인코딩을 걷어내고 프레임워크에 맡기는 수정을 올렸습니다.

8분 뒤인 5시 42분에 두 번째 커밋이 올라갔습니다. 이번에는 공백이 들어간 태그가 문제였습니다. "모든 개발자를 위한 HTTP 웹 기본 지식"이라는 긴 태그의 공백이 주소에서 %20으로 바뀌어 그대로 노출됐고, 일단 태그 이름을 "HTTP 웹 기본"으로 줄여서 피했습니다.

여기서부터 12분 동안 설계가 세 번 뒤집혔습니다. 5시 46분에는 한글 태그를 영문 주소로 바꿔주는 매핑(학습은 study로, 비동기는 async로)을 만들어 근본 해결을 시도했고, 5시 48분에는 줄여놨던 태그 이름을 원래대로 되돌렸습니다. 5시 50분에는 그 매핑을 도로 걷어내고 한글 주소를 그대로 쓰는 쪽으로 갔다가, 5시 54분에 다시 매핑을 복구했습니다. 최종 형태는 주소는 영문(/tags/study), 화면에 보이는 제목은 한글(#학습)로 역할을 나누는 것이었습니다.

![배포 당일 17시 34분부터 54분까지 20분 사이에 연달아 쌓인 태그 수정 커밋 여섯 개의 git log](/images/blog-rebuild-retrospective/tag-fix-commits.png)

첫 수정부터 마지막 수정까지 20분 동안 커밋 여섯 개가 쌓였고, 같은 코드가 생겼다 사라졌다 다시 생겼습니다. 돌아보면 왕복의 원인은 주소에 쓸 값과 화면에 보여줄 값을 하나의 값으로 다루려 한 데 있었습니다. 주소는 인코딩 문제가 없어야 하고 화면 제목은 원래 이름 그대로여야 하는데, 이 두 요구는 한글 태그에서는 한 값으로 동시에 만족되지 않습니다. 두 값을 분리하고 나서야 왕복이 끝났습니다.

저녁에는 태그 페이지들을 sitemap에 추가하고(6시 54분) Google Search Console 소유권 인증 메타태그를 붙이는(6시 56분) 마무리 작업으로 배포 당일을 정리했습니다.

![이식이 끝난 블로그 홈 화면. 검색창 아래로 옮겨 온 포스트 목록이 이어집니다](/images/blog-rebuild-retrospective/blog-home.png)

## 배운 것

첫째, 버전 관리는 작업 첫날 시작했어야 합니다. 이 회고를 쓰려고 기록을 뒤졌을 때 시각 단위로 되짚을 수 있는 건 배포 당일 저녁뿐이었습니다. 이식 과정 전체가 19,356줄짜리 첫 커밋 하나에 뭉쳐 있어서, 어떤 글에서 어떤 시행착오를 겪었는지는 프로젝트 문서에 따로 적어둔 결정 기록으로만 남았습니다. 심지어 그 문서의 할 일 목록에 있던 "Git 저장소 초기화" 항목은 지워지지 않은 채로 첫 커밋에 함께 들어갔습니다. 과정의 기록은 나중에 만들 수 없습니다. 그때그때 커밋으로 남기지 않으면 회고를 쓸 재료 자체가 사라집니다.

둘째, 플랫폼 위에 발행한 글은 통째로 내 것이 아닙니다. 서비스가 종료된 블로그의 본문은 Wayback Machine 덕에 살아남았지만 이미지 18장은 어디에도 남지 않았습니다. 아카이브 서비스는 페이지 위주로 저장합니다. 이미지 같은 첨부 자산은 훨씬 쉽게 유실됩니다. 그래서 이번 이식에서는 글에 딸린 이미지와 영상을 전부 저장소 안의 정적 파일로 옮겼습니다. 발행하는 곳과 별개로 원본 자산을 직접 들고 있어야, 다음에 어떤 서비스가 문을 닫아도 같은 일을 다시 겪지 않습니다.

셋째, 요구가 다른 두 값은 처음부터 분리해야 합니다. 이중 인코딩 버그도, 그 뒤 20분의 설계 왕복도 뿌리는 같았습니다. 주소에 들어갈 값과 사람에게 보여줄 값은 요구하는 성질이 다른데 이를 값 하나로 해결하려 했던 것입니다. 인코딩을 직접 할지 프레임워크에 맡길지도 마찬가지로 한쪽으로 정해야 합니다. 같은 값을 두 주체가 손대는 순간 %ED%95%99 같은 문자열이 화면에 나타납니다.

---

이제 글 42편은 저장소 안의 MDX 파일 42개와 이미지 폴더로 존재합니다. 다음에 또 블로그를 옮기게 되더라도 이 폴더를 복사하는 것으로 이식이 끝납니다. 원자료를 특정 서비스 안이 아니라 파일로 소유하고, 보여주는 쪽만 갈아 끼우는 구조는 글이든 사진이든 오래 가져갈 기록 전반에 적용할 수 있습니다. 서비스는 언젠가 종료되고, 그 뒤에 확실히 남아 있는 것은 파일로 들고 있던 것뿐이라는 사실을 이번 이식에서 확인했습니다.
